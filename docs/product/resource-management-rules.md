## 1. 文档目标

本文档定义平台内“资源”的管理规则，作为产品、后端、小程序、管理后台和运营执行的统一依据。

资源管理的业务目标是：

- 让真实有效的服装产业资源被快速发布、审核、搜索、展示和联系。
- 让货源、库存、工厂产能、订单、招聘、出租和服务复用同一套发布、审核、生命周期、搜索、联系、统计和运营能力。
- 通过认证、信用、效果数据和人工撮合，逐步沉淀可信的产业资源库。
- 支持织里首站验证后复制到广州、虎门、杭州、常熟、石狮等产业带。

本文档不是接口字段全集，也不是数据库设计全集。接口、表结构、字段类型以 API 契约和数据库设计文档为准；本文档负责明确业务规则和判断标准。

## 2. 核心原则

### 2.1 资源是统一业务对象

所有可被发布、审核、搜索、展示、联系、撮合和统计效果的供给或需求，默认都应建模为资源。

第一阶段资源类型包括：

- `inventory`：库存资源。
- `goods`：货源资源。
- `factory`：工厂产能资源。
- `order`：订单需求资源。
- `job`：招聘资源。
- `rental`：出租/转让资源。
- `service`：物流、辅料、印花、绣花、摄影、直播等服务资源。

除非业务存在完全不同的交易、履约、结算或权限模型，否则不得为某一类资源单独复制一套发布、审核、搜索、消息、联系统计和后台系统。

### 2.2 配置优先于新系统

新增业务时按以下顺序判断：

1. 是否可以新增资源类型承接。
2. 是否只需要新增结构化字段。
3. 是否只需要新增筛选字段或展示模板。
4. 是否只需要调整有效期、审核规则、排序权重或消息规则。
5. 是否真的需要独立系统。

只有当前四项无法覆盖业务差异，并且该业务具备独立交易、履约、结算或复杂权限边界时，才允许立项独立系统。

### 2.3 真实性优先于数量

平台不能为了资源数量放松真实性要求。MVP 阶段宁可资源量少，也要保证：

- 联系方式真实可用。
- 标题、图片、价格、数量、产能等核心信息与资源类型匹配。
- 审核结论有记录，驳回和下架有原因。
- 用户可见信息不夸大、不误导、不伪造成交效果。

### 2.4 生命周期必须可追踪

每条资源从创建到归档必须有清晰状态。任何公开展示、联系、刷新、置顶、标记成交、下架和归档行为，都必须受状态约束。

资源状态不得只依赖前端判断。后端必须以资源当前状态作为最终判断依据。

### 2.5 商家主页承接长期信任

资源是单次供需信息，商家是长期经营主体。所有资源必须归属商家，公开详情应能进入商家主页，让用户判断商家的认证状态、信用标签、历史发布和活跃情况。

## 3. 统一语言

| 术语 | 说明 |
|---|---|
| 资源 | 可被搜索、展示、联系和撮合的供给或需求，是平台资源管理的核心对象。 |
| 资源类型 | 资源的业务分类，如库存、货源、工厂产能、订单、招聘、出租、服务。 |
| 资源类型配置 | 定义某类资源的字段、必填项、筛选项、展示模板、有效期、审核规则、排序权重和消息规则。 |
| 商家 | 工厂、档口、库存商、服务商、采购商等经营主体。 |
| 商家管理员 | 有权代表商家发布、编辑、刷新、下架资源并查看效果数据的用户。 |
| 平台运营 | 有权审核、驳回、下架、调整配置和进行人工撮合的后台用户。 |
| 公开资源 | 状态为已发布、未过期、未下架、未归档，且所属商家可正常展示的资源。 |
| 认证资源 | 经过平台核验并有认证记录或信用记录背书的资源。 |
| 发布效果 | 资源曝光、详情浏览、电话点击、微信复制、商家主页访问、分享、成交反馈等数据。 |

## 4. 资源类型规则

### 4.1 库存资源 `inventory`

库存资源用于解决尾货、断码断色、换季库存、仓库积压和整包出清问题。

核心字段：

- 城市/产业带。
- 品类。
- 性别/年龄段。
- 季节。
- 尺码段。
- 数量。
- 打包价或单件参考价。
- 是否混批。
- 是否支持拿样。
- 是否支持直播。
- 仓库位置。
- 图片或视频。
- 联系方式。

管理规则：

- 库存数量、尺码段、价格描述必须尽量具体，不能只写“很多”“便宜”“详聊”。
- 图片应优先展示实物、整包、仓库或吊牌，不得使用与实际库存无关的网图冒充。
- 已售罄、已处理或长期无人响应的库存应及时标记成交、下架或过期。
- 平台核实过的库存可以展示“已认证库存”“平台已核实”等信用标签。

默认有效期建议：7 天。

### 4.2 货源资源 `goods`

货源资源用于解决门店、批发商、电商卖家、主播找源头货、爆款、新款和稳定供货的问题。

核心字段：

- 城市/产业带。
- 品类。
- 风格。
- 价格带。
- 起批量。
- 是否现货。
- 是否一件代发。
- 是否支持直播供货。
- 是否可退换。
- 发货时效。
- 商品图片。
- 商家认证状态。

管理规则：

- 货源标题应体现品类、风格或供货优势，不能使用泛化标题。
- 如宣传“一件代发”“直播供货”“现货”，审核时应检查字段和描述是否一致。
- 货源图片应能让采购方判断款式，不得用纯文字海报替代实物展示。
- 长期稳定货源可适当设置更长有效期，但仍需通过刷新体现近期活跃。

默认有效期建议：15 天。

### 4.3 工厂产能资源 `factory`

工厂产能资源用于解决找加工厂、找空档产能、小单快反、包工包料和配套加工问题。

核心字段：

- 城市/产业带。
- 工厂类型。
- 擅长品类。
- 日产能。
- 起订量。
- 是否接小单。
- 是否包工包料。
- 当前空档期。
- 工价范围。
- 工厂人数。
- 设备情况。
- 工厂实拍。
- 认证状态。

管理规则：

- 工厂产能必须描述可承接的品类、数量、排期或加工方式。
- “小单快反”“包工包料”“日产能”等能力描述应尽量结构化，避免只写宣传语。
- 工厂实拍应展示车间、设备、样衣、工位或门头，不得只放商品图。
- 已认证工厂可在搜索和展示中获得可信度加权。

默认有效期建议：30 天。

### 4.4 订单需求资源 `order`

订单需求资源用于订单方、品牌方、电商卖家或批发商寻找可承接订单的工厂或服务商。

核心字段：

- 可接或待找品类。
- 数量范围。
- 交期要求。
- 报价方式。
- 是否支持打样。
- 是否支持小单快反。
- 是否长期合作。
- 质量或工艺要求。
- 联系方式。

管理规则：

- 订单需求应说明品类、数量、交期和合作方式，避免只写“有订单找工厂”。
- 涉及明确预算或价格区间时，应以区间表达，避免诱导性极低价格。
- 订单需求可进入人工撮合池，由运营匹配工厂、服务商或库存资源。
- 已完成对接或不再需要的订单需求应标记成交或关闭。

默认有效期建议：15 天。

### 4.5 招聘资源 `job`

招聘资源用于工厂、档口、仓库和服务商发布招工需求。

核心字段：

- 岗位。
- 工价或薪资范围。
- 人数。
- 是否包吃住。
- 工作地点。
- 结算方式。
- 工作时长或班次。
- 联系方式。

管理规则：

- 招聘资源必须包含岗位、人数、地点和结算方式中的核心信息。
- 工价、薪资和福利不得夸大，不得发布明显虚假高薪信息。
- 招聘属于辅助资源类型，不应影响平台“找货、找厂、清库存、接订单”的主线权重。
- 超过有效期仍未刷新时，默认认为岗位信息可能失效。

默认有效期建议：15 天。

### 4.6 出租/转让资源 `rental`

出租/转让资源用于厂房、档口、仓库、设备或经营位置的出租、转租、转让。

核心字段：

- 位置。
- 面积。
- 租金。
- 楼层。
- 用途。
- 是否可住。
- 转让费。
- 图片。
- 联系方式。

管理规则：

- 出租/转让资源必须描述位置、面积、价格或转让条件。
- 图片应体现空间、门头、楼层、设备或周边环境。
- 涉及转让费、押金、租期等敏感条件时，必须避免误导性表述。
- 已出租或已转让后应及时标记成交或下架。

默认有效期建议：30 天。

### 4.7 服务资源 `service`

服务资源用于物流、辅料、印花、绣花、摄影、直播、后道、包装等服装产业配套服务。

核心字段：

- 服务类型。
- 服务范围。
- 可服务城市或区域。
- 价格方式。
- 响应时效。
- 服务案例。
- 图片。
- 联系方式。

管理规则：

- 服务资源必须明确服务内容，不能只写公司介绍。
- 可展示服务案例、设备、团队或现场图片，但不得冒用他人案例。
- 认证服务商可展示“已认证服务商”标签。
- 服务资源可作为撮合候选资源参与订单、货源和工厂需求匹配。

默认有效期建议：30 天。

## 5. 资源归属与权限规则

### 5.1 资源必须归属商家

每条资源必须绑定一个商家。没有商家主体的用户不能直接发布公开资源。

允许的发布主体：

- 商家管理员代表所属商家发布。
- 平台运营代商家发布，资源仍归属被代发商家。

不允许的发布方式：

- 普通用户绕过商家直接发布公开资源。
- 运营将资源发布到不存在、停用或被封禁的商家名下。
- 一个资源同时归属多个商家。

### 5.2 商家状态约束

商家状态影响资源管理动作：

| 商家状态 | 资源创建 | 资源提交审核 | 公开展示 | 联系动作 | 后台操作 |
|---|---|---|---|---|---|
| `active` 正常 | 允许 | 允许 | 允许 | 允许 | 允许 |
| `suspended` 暂停 | 可限制 | 禁止或需运营确认 | 可下架 | 可限制 | 允许 |
| `disabled` 停用 | 禁止 | 禁止 | 禁止 | 禁止 | 仅允许查看和审计 |

当商家被停用时，其公开资源应从前台搜索和详情中隐藏，但历史审核、联系、统计和日志记录不得物理删除。

### 5.3 用户权限边界

商家管理员可以：

- 创建所属商家的资源。
- 编辑草稿、驳回、已过期、已下架资源。
- 提交审核。
- 查看我的资源列表和效果数据。
- 对已发布资源执行刷新、置顶、标记成交、主动下架。

商家管理员不可以：

- 审核自己的资源。
- 修改审核记录。
- 修改其他商家的资源。
- 强制恢复已被平台下架的资源。
- 修改资源类型配置。

平台运营可以：

- 查看管理范围内城市站资源。
- 审核、驳回、下架资源。
- 代商家发布或修正资源。
- 维护资源类型配置。
- 查看资源效果、联系事件和操作日志。
- 将资源加入专题、榜单或人工撮合。

平台运营不可以：

- 绕过日志执行敏感操作。
- 删除关键业务记录。
- 伪造成交反馈或虚假效果数据。
- 将风险资源包装为认证资源。

## 6. 字段管理规则

### 6.1 稳定字段必须关系化

以下字段属于所有资源共享的稳定字段，应作为 `resources` 的关系化字段管理：

- 所属商家。
- 城市站。
- 资源类型配置。
- 资源类型编码。
- 状态。
- 标题。
- 品类。
- 区域。
- 价格描述。
- 数量/产能描述。
- 封面。
- 描述。
- 标签。
- 图片。
- 联系人。
- 联系电话。
- 联系微信。
- 是否认证。
- 发布时间。
- 刷新时间。
- 过期时间。
- 成交时间。
- 下架时间。
- 归档时间。
- 驳回原因。
- 下架原因。

这些字段用于权限、搜索、筛选、列表、详情、生命周期、统计和后台运营，不应随意迁移到扩展字段。

### 6.2 类型差异字段进入 `attributes`

服装产业字段变化快，类型差异字段默认进入 `attributes JSONB`，由资源类型配置统一管理。

示例：

```json
{
  "season": "春款",
  "sizeRange": "90-140",
  "allowLiveSale": true,
  "allowSample": true,
  "packagePrice": "18元/件"
}
```

当某个扩展字段满足以下任一条件时，可以评估是否冗余为关系化字段或增加表达式索引：

- 频繁用于搜索筛选。
- 高频参与排序。
- 需要强一致校验。
- 需要运营后台批量统计。
- 查询性能已经成为实际瓶颈。

### 6.3 必填字段规则

资源发布时必须校验：

- 通用必填字段：商家、城市站、资源类型、标题、品类、描述、联系人、联系电话。
- 类型必填字段：由 `resource_type_configs.required_fields` 决定。
- 联系方式至少应包含可用电话；微信可选，但填写后必须符合平台格式规则。
- 图片是否必填由资源类型配置决定。库存、货源、工厂、出租、服务原则上应至少有一张图片。

缺失必填字段时，接口必须返回中文友好提示，不能返回内部字段名或数据库错误。

### 6.4 标题规则

标题应帮助用户快速判断资源价值。

标题建议包含：

- 品类。
- 资源类型关键点。
- 价格、数量、产能、区域或时效中的一个核心卖点。

标题禁止：

- 只有“急出”“好货”“招人”“联系我”等泛化词。
- 使用明显夸大、误导或虚假承诺。
- 带无关城市、无关品类或无关平台导流词。
- 包含联系方式。联系方式应进入联系字段，不应写在标题里。

### 6.5 图片规则

图片是服装产业资源真实性的重要依据。

通用规则：

- 列表缩略图建议使用 1:1。
- 详情主图建议使用 4:3。
- 优先展示真实货品、工厂、档口、仓库、设备、场地或服务案例。
- 缺图时可使用统一占位图，但不应把占位图包装为真实图片。

禁止图片：

- 与资源无关的网图。
- 明显盗用其他商家图片。
- 纯二维码、纯联系方式或纯广告海报。
- 涉及违法违规、低俗或侵权内容。

### 6.6 联系方式规则

资源联系方式用于真实撮合，不用于公开爬取。

规则：

- 电话必须脱敏展示或通过联系动作触发后展示，具体展示方式由前端和风控策略决定。
- 微信复制、电话点击、进入商家主页和分享都应记录联系事件。
- 联系字段变更属于敏感变更，已发布资源修改后原则上需要重新审核。
- 被投诉联系方式无效或骚扰时，平台可下架资源并记录风险标签。

## 7. 生命周期规则

### 7.1 状态定义

资源状态使用以下稳定枚举：

| 状态 | 中文名 | 定义 | 是否公开展示 |
|---|---|---|---|
| `draft` | 草稿 | 已创建但未提交审核。 | 否 |
| `pending` | 待审核 | 已提交审核，等待平台处理。 | 否 |
| `published` | 已发布 | 审核通过，未过期，未下架。 | 是 |
| `rejected` | 已驳回 | 审核未通过，需修改后重新提交。 | 否 |
| `expired` | 已过期 | 超过有效期，需刷新、重发或归档。 | 否 |
| `dealt` | 已成交 | 商家或运营标记资源已完成对接或强意向成交。 | 否 |
| `taken_down` | 已下架 | 商家主动下架或平台强制下架。 | 否 |
| `archived` | 已归档 | 资源进入历史归档，仅保留记录。 | 否 |

公开列表、搜索、专题和详情默认只返回 `published` 且 `expires_at` 晚于当前时间的资源。

### 7.2 状态流转

允许的核心流转：

| 当前状态 | 允许动作 | 目标状态 | 操作主体 |
|---|---|---|---|
| `draft` | 提交审核 | `pending` | 商家管理员、平台运营 |
| `draft` | 归档 | `archived` | 商家管理员、平台运营 |
| `pending` | 审核通过 | `published` | 平台运营 |
| `pending` | 审核驳回 | `rejected` | 平台运营 |
| `pending` | 平台下架 | `taken_down` | 平台运营 |
| `rejected` | 修改后重新提交 | `pending` | 商家管理员、平台运营 |
| `published` | 编辑关键字段 | `pending` | 商家管理员、平台运营 |
| `published` | 刷新 | `published` | 商家管理员、平台运营 |
| `published` | 标记成交 | `dealt` | 商家管理员、平台运营 |
| `published` | 主动下架 | `taken_down` | 商家管理员、平台运营 |
| `published` | 到期任务处理 | `expired` | 系统任务 |
| `expired` | 复制再发 | `draft` | 商家管理员、平台运营 |
| `expired` | 归档 | `archived` | 商家管理员、平台运营、系统任务 |
| `dealt` | 复制再发 | `draft` | 商家管理员、平台运营 |
| `dealt` | 归档 | `archived` | 商家管理员、平台运营、系统任务 |
| `taken_down` | 修改后重新提交 | `pending` | 商家管理员、平台运营 |
| `taken_down` | 归档 | `archived` | 平台运营、系统任务 |

禁止流转：

- `pending` 直接公开展示。
- `rejected` 直接变为 `published`。
- `expired` 直接变为 `published`，除非通过重新提交审核或明确的续期规则。
- `taken_down` 被商家自行恢复为 `published`。
- `archived` 恢复为公开资源。

### 7.3 审核通过规则

审核通过时必须：

- 将状态置为 `published`。
- 写入 `published_at`。
- 按资源类型配置写入 `expires_at`。
- 写入审核记录，包含审核人、动作、原因或备注、资源快照。
- 触发审核通过消息。

如果资源之前已经发布过，重新审核通过时应更新 `published_at` 或保留首次发布时间的策略必须在实现中保持一致。MVP 建议重新审核通过时更新最近发布时间，并保留历史审核记录用于追踪。

### 7.4 驳回规则

资源存在以下情况应驳回：

- 必填字段缺失或明显不完整。
- 标题、描述、图片和资源类型不匹配。
- 联系方式缺失、格式异常或明显不可用。
- 图片不真实、与资源无关或无法证明资源存在。
- 价格、数量、产能、工资等关键内容明显夸大或诱导。
- 存在违法违规、侵权、低俗或恶意导流内容。
- 所属商家异常，无法确认发布主体。

驳回时必须填写中文驳回原因，并尽量给出可修改方向。

示例：

- `请补充库存数量、尺码段和实物图片后重新提交`
- `工厂产能信息不完整，请补充擅长品类、日产能和工厂实拍`
- `联系方式无法核实，请确认电话或微信后重新提交`

### 7.5 下架规则

下架分为商家主动下架和平台强制下架。

商家主动下架适用于：

- 资源已售罄或已对接。
- 信息暂时不想公开。
- 发布内容需要重做。

平台强制下架适用于：

- 被证实虚假。
- 被多次投诉且无法解释。
- 联系方式无效或存在骚扰。
- 涉及违规内容。
- 商家被停用或封禁。
- 资源过期后仍通过异常方式公开。

平台强制下架必须记录下架原因和操作日志。商家不得自行恢复平台强制下架资源，只能修改后重新提交审核。

### 7.6 过期规则

资源有效期由资源类型配置决定。系统任务应定期处理过期资源：

- 已发布资源到达 `expires_at` 后变更为 `expired`。
- 过期资源不进入公开列表、搜索、专题或推荐。
- 过期前可发送即将过期消息。
- 过期后可发送已过期消息。
- 商家可以复制过期资源再发布，新资源从 `draft` 开始。

MVP 建议默认有效期：

| 资源类型 | 默认有效期 |
|---|---:|
| 库存 | 7 天 |
| 货源 | 15 天 |
| 工厂产能 | 30 天 |
| 订单需求 | 15 天 |
| 招聘 | 15 天 |
| 出租/转让 | 30 天 |
| 服务 | 30 天 |

城市站可在资源类型配置中覆盖默认有效期。

### 7.7 归档规则

归档用于降低活跃数据干扰，但不删除历史记录。

可归档资源：

- 长期过期未处理资源。
- 已成交且无需继续展示资源。
- 已下架且无继续申诉或修改价值资源。
- 草稿长期未提交资源。

归档后：

- 不允许公开展示。
- 不允许刷新、置顶、标记成交。
- 可在后台或商家历史记录中查看。
- 不得物理删除资源、审核记录、联系事件和效果统计。

## 8. 发布与编辑规则

### 8.1 创建资源

创建资源时必须完成：

- 校验用户是否有商家发布权限。
- 校验商家状态是否允许发布。
- 校验城市站是否启用。
- 校验资源类型配置是否启用。
- 校验通用必填字段和类型必填字段。
- 将通用字段写入稳定字段。
- 将类型差异字段写入 `attributes`。

创建后可以直接进入 `pending`，也可以先保存为 `draft`。具体入口规则：

- 小程序快速发布：信息完整时建议直接提交审核。
- 后台运营代发：允许保存草稿或直接提交审核。
- 复杂资源：允许先保存草稿，再补充图片或字段后提交。

### 8.2 编辑资源

不同状态下编辑规则不同：

| 状态 | 是否可编辑 | 编辑后状态 |
|---|---|---|
| `draft` | 可编辑 | 保持 `draft` |
| `pending` | 不建议编辑；如允许，需重新提交审核 | 保持或重置为 `pending` |
| `published` | 可编辑，但关键字段变更需重新审核 | `pending` 或保持 `published` |
| `rejected` | 可编辑 | `rejected`，提交后为 `pending` |
| `expired` | 不直接编辑公开版本，建议复制再发 | 新草稿 |
| `dealt` | 不直接编辑公开版本，建议复制再发 | 新草稿 |
| `taken_down` | 商家主动下架可编辑后提交；平台强制下架需重新审核 | `pending` |
| `archived` | 不可编辑 | 保持 `archived` |

关键字段包括：

- 标题。
- 品类。
- 描述。
- 价格。
- 数量或产能。
- 图片。
- 联系方式。
- 资源类型差异字段中的重要字段。

非关键字段包括：

- 标签。
- 局部格式修正。
- 运营备注。
- 不影响用户判断的展示文案。

MVP 建议采用保守策略：已发布资源修改关键内容后统一重新进入审核。

### 8.3 复制再发

复制再发用于过期、成交或下架资源快速重发。

规则：

- 复制后生成新的资源 ID。
- 新资源状态为 `draft`。
- 不复制原资源的审核记录、联系事件、效果统计、置顶状态和成交状态。
- 可复制标题、品类、描述、图片、联系信息和 `attributes`。
- 新资源提交后必须重新审核。

## 9. 审核管理规则

### 9.1 审核队列

进入审核队列的资源包括：

- 新提交资源。
- 驳回后重新提交资源。
- 已发布资源关键字段变更后重新提交。
- 平台认为需要复核的资源。

审核列表应支持：

- 城市站筛选。
- 资源类型筛选。
- 品类筛选。
- 提交时间排序。
- 商家认证状态展示。
- 历史驳回原因查看。

### 9.2 审核标准

审核员应按以下顺序判断：

1. 发布主体是否可信：商家是否存在、状态是否正常、发布人是否有权限。
2. 资源类型是否匹配：内容是否属于所选类型。
3. 核心字段是否完整：用户能否凭这些信息做初步判断。
4. 图片是否有效：是否能证明货品、工厂、场地或服务存在。
5. 联系方式是否可用：是否明显缺失、异常或无效。
6. 是否存在违规内容：虚假、侵权、导流、低俗、违法等。
7. 是否需要认证或风险标签：对真实性要求高的资源可建议认证。

### 9.3 审核记录

每次审核必须写入审核记录：

- 资源 ID。
- 审核人。
- 审核动作：通过、驳回、下架。
- 原因或备注。
- 审核时资源快照。
- 审核时间。

审核记录用于：

- 追踪资源状态变化。
- 支撑商家申诉。
- 复盘运营质量。
- 防止敏感操作无记录。

### 9.4 审核消息

审核动作应触发消息：

- 审核通过：告诉商家资源已发布，并提示有效期。
- 审核驳回：告诉商家驳回原因和修改方向。
- 平台下架：告诉商家下架原因和是否可重新提交。

消息文案必须中文、明确、可操作，不得只展示内部状态码。

## 10. 公开展示规则

### 10.1 公开可见条件

资源公开可见必须同时满足：

- 资源状态为 `published`。
- 当前时间未超过 `expires_at`。
- 所属商家状态正常。
- 所属城市站启用。
- 资源类型配置启用。
- 资源未被风控强制隐藏。

任何一个条件不满足，资源不得出现在：

- 首页资源列表。
- 搜索结果。
- 专题列表。
- 同类推荐。
- 商家主页当前资源。
- 资源详情公开访问。

### 10.2 列表展示

列表卡片应优先展示用户决策字段：

- 标题。
- 类型。
- 品类。
- 价格描述。
- 数量/产能描述。
- 城市或区域。
- 商家名称。
- 认证状态。
- 最近刷新时间。
- 封面图。

不同资源类型可以通过展示模板调整重点字段。

### 10.3 详情展示

详情页应展示：

- 资源核心字段。
- 类型差异字段。
- 图片或视频。
- 商家入口。
- 认证和信用标签。
- 联系动作。
- 同类推荐。
- 举报入口。

详情页不能展示：

- 待审核、驳回、过期、下架、归档资源的公开详情。
- 被平台隐藏的联系方式。
- 内部审核备注、风控备注、操作日志。

### 10.4 商家主页中的资源展示

商家主页应区分：

- 当前资源：公开可见的已发布资源。
- 历史资源：已过期、已成交或已归档资源，可根据产品策略展示部分摘要。

商家主页展示历史资源时不得让用户误以为仍可联系该资源。历史资源应明确状态。

## 11. 搜索、筛选和排序规则

### 11.1 搜索范围

搜索默认只检索公开资源。后台检索可以按权限查看全部状态资源。

前台搜索条件包括：

- 关键词。
- 城市站。
- 资源类型。
- 品类。
- 价格区间。
- 数量/产能区间。
- 认证状态。
- 更新时间。
- 类型配置中的筛选字段。

### 11.2 筛选字段

筛选字段由资源类型配置决定。每个筛选字段必须满足：

- 有清晰中文标签。
- 有稳定字段编码。
- 前端可以渲染。
- 后端可以解析和过滤。
- 不暴露内部字段结构。

示例：

- 库存：季节、尺码段、是否支持直播、是否支持拿样。
- 工厂：擅长品类、日产能、是否接小单、是否包工包料。
- 招聘：岗位、工价、是否包吃住、结算方式。
- 出租：面积、租金、用途、是否可住。

### 11.3 排序规则

MVP 默认排序建议：

1. 置顶资源优先。
2. 已认证资源优先。
3. 最近刷新资源优先。
4. 图片和核心字段完整度更高的资源优先。
5. 投诉或风险资源降权。

不得把已过期、已下架、已驳回资源通过排序规则重新放回公开结果。

### 11.4 搜索无结果

搜索无结果时应：

- 展示提交采购需求入口。
- 记录搜索词、筛选条件和结果数。
- 将高频无结果搜索反馈给运营，用于供给增长。

不得用无关资源填充结果页，避免误导用户。

## 12. 刷新与置顶规则

### 12.1 刷新资源

刷新用于表达资源近期有效，而不是绕过审核。

允许刷新条件：

- 资源状态为 `published`。
- 资源未过期。
- 所属商家状态正常。
- 商家有刷新额度或符合免费刷新规则。

刷新后：

- 更新 `refreshed_at`。
- 影响搜索排序。
- 记录操作日志或权益流水。
- 不改变审核状态。

禁止刷新：

- 待审核资源。
- 驳回资源。
- 已过期资源。
- 已成交资源。
- 已下架资源。
- 已归档资源。
- 平台限制商家的资源。

### 12.2 置顶资源

置顶用于商业化和运营推荐，必须受状态和权益约束。

允许置顶条件：

- 资源状态为 `published`。
- 资源未过期。
- 资源归属当前商家。
- 置顶券未使用、未过期、未作废。
- 置顶券允许用于该资源类型。

置顶后：

- 记录置顶券核销。
- 记录置顶开始时间和结束时间，或按当前数据模型记录使用资源和时长。
- 影响排序权重。
- 不改变资源状态。

禁止置顶：

- 待审核、驳回、过期、成交、下架、归档资源。
- 非当前商家资源。
- 风险资源。
- 已过期或已使用置顶券。

### 12.3 置顶不改变真实性责任

置顶只改变展示优先级，不代表平台认证。未认证资源置顶时，不得展示为认证资源。

## 13. 联系与效果统计规则

### 13.1 联系事件

以下动作应记录为联系事件：

- 电话点击。
- 微信复制。
- 进入商家主页。
- 分享资源。

联系事件应记录：

- 资源 ID。
- 联系用户 ID，允许匿名为空。
- 资源所属商家 ID。
- 动作类型。
- 发生时间。

### 13.2 效果统计

资源效果数据包括：

- 曝光。
- 搜索曝光。
- 详情浏览。
- 电话点击。
- 微信复制。
- 商家主页访问。
- 收藏。
- 分享。
- 成交反馈。

MVP 至少应支持：

- 详情浏览。
- 电话点击。
- 微信复制。
- 商家主页访问或分享中的基础联系事件。

### 13.3 统计展示原则

效果数据用于帮助商家判断发布质量，不能包装成虚假成交。

展示原则：

- 明确区分曝光、浏览、联系和成交反馈。
- 不展示无法证明的成交额。
- 不伪造平台推荐效果。
- 不为了营销夸大资源热度。

### 13.4 幂等与反作弊

统计实现应考虑：

- 同一用户短时间重复浏览可以合并或限频。
- 联系动作可以按真实点击记录，但后台展示时可按天聚合。
- 异常高频操作应进入风控或降权判断。
- 商家自己浏览自己的资源不应被包装为真实买家意向。

## 14. 成交反馈规则

### 14.1 标记成交

标记成交用于表达资源已经达成交易、完成对接或形成明确强意向，不等同于平台担保交易。

允许标记成交条件：

- 资源状态为 `published`。
- 操作者为资源所属商家管理员或平台运营。

标记成交后：

- 状态变更为 `dealt`。
- 写入 `dealt_at`。
- 不再公开展示。
- 可生成信用记录或成交反馈记录。
- 可用于商家主页信用标签展示，但必须避免夸大。

### 14.2 成交反馈真实性

成交反馈应区分来源：

- 商家自报。
- 买家反馈。
- 平台核实。

只有平台核实或多方确认的成交反馈，才适合转化为公开信用标签。

### 14.3 已成交资源处理

已成交资源可以：

- 进入商家历史记录。
- 用于商家复盘效果。
- 复制再发类似资源。
- 作为信用记录依据。

已成交资源不可以：

- 继续公开当作可联系资源。
- 继续刷新或置顶。
- 被直接恢复为已发布。

## 15. 认证与信用规则

### 15.1 认证对象

认证可以作用于：

- 商家：认证工厂、认证档口、认证服务商。
- 资源：认证库存、平台核实资源。
- 服务能力：物流、辅料、印花、绣花、摄影、直播等服务能力。

### 15.2 认证和资源展示关系

认证通过后可以影响：

- 资源详情认证标签。
- 商家主页认证状态。
- 搜索筛选和排序权重。
- 运营推荐和人工撮合优先级。

认证不得影响：

- 已驳回资源直接变为已发布。
- 已下架资源自动恢复。
- 过期资源重新公开。

认证是可信背书，不是生命周期豁免。

### 15.3 信用标签规则

公开信用标签必须有事实依据。

允许公开展示的标签示例：

- 已认证工厂。
- 已认证档口。
- 已认证库存。
- 已认证服务商。
- 平台已核实。
- 近期活跃。
- 多次发布。
- 有成交反馈。

风险类标签展示要谨慎，适合按产品策略区分公开和内部：

- 有投诉记录。
- 联系方式异常。
- 信息待复核。
- 风险提示。

### 15.4 信用记录撤销

当认证失效、投诉成立或平台复核发现问题时，应撤销或更新相关信用记录。

规则：

- 不物理删除历史信用记录。
- 使用撤销时间或状态表达失效。
- 公开标签只基于当前有效且可公开的信用记录生成。

## 16. 风控、举报和违规处理

### 16.1 举报入口

资源详情应提供举报入口。举报类型包括：

- 虚假信息。
- 联系不上。
- 价格或数量不实。
- 图片侵权或盗图。
- 违法违规内容。
- 恶意导流。
- 其他。

### 16.2 风险处理

平台收到举报后可以：

- 标记内部风险。
- 降低排序权重。
- 要求商家补充材料。
- 临时隐藏资源。
- 强制下架资源。
- 限制商家发布。
- 记录信用风险。

### 16.3 高风险资源

高风险资源包括：

- 多次被投诉且无法解释。
- 联系方式长期无效。
- 图片、价格、数量与实际严重不符。
- 诱导用户离开平台进行不安全交易。
- 涉嫌违法违规或侵权。

高风险资源应优先下架，不得进入专题、榜单、置顶或人工推荐。

## 17. 消息规则

资源管理相关消息包括：

- 审核通过。
- 审核驳回。
- 资源即将过期。
- 资源已过期。
- 资源被下架。
- 资源被联系。
- 资源效果反馈。
- 撮合进度更新。
- 认证结果。

消息触发原则：

- 对用户有明确下一步动作时才发。
- 文案必须中文、明确、可操作。
- 不做营销群发。
- 不因无意义曝光波动频繁打扰商家。

## 18. 运营后台规则

### 18.1 后台资源列表

后台资源列表应支持：

- 城市站筛选。
- 资源类型筛选。
- 状态筛选。
- 商家筛选。
- 关键词搜索。
- 认证状态筛选。
- 发布时间、刷新时间、过期时间排序。

列表应展示：

- 标题。
- 商家。
- 资源类型。
- 城市站。
- 状态。
- 提交时间。
- 发布时间。
- 过期时间。
- 认证状态。
- 基础效果数据。

### 18.2 后台敏感操作

敏感操作包括：

- 审核通过。
- 审核驳回。
- 平台下架。
- 修改资源类型配置。
- 修改商家状态。
- 代商家发布。
- 修改认证和信用标签。
- 发放或核销权益。

敏感操作必须记录操作日志，包括操作人、对象、动作、原因、时间和必要快照。

### 18.3 人工撮合

资源可以进入人工撮合流程。运营创建撮合记录时应：

- 关联采购需求或搜索无结果线索。
- 添加候选资源。
- 添加参与方。
- 记录联系进展。
- 记录撮合结果。

只有公开有效或经运营确认可对接的资源，才应作为候选资源推荐给需求方。

## 19. 城市站扩展规则

### 19.1 城市站继承统一模型

新增城市站时，不允许复制一套独立资源系统。城市差异通过以下配置实现：

- 城市站启用状态。
- 城市站可用资源类型。
- 品类配置。
- 字段模板。
- 筛选字段。
- 展示模板。
- 有效期。
- 审核规则。
- 排序权重。
- 专题和榜单。

### 19.2 城市差异示例

织里站重点：

- 童装库存。
- 童装货源。
- 小单快反工厂。
- 招工。
- 仓库或厂房出租。

广州站重点：

- 女装新款。
- 档口货源。
- 直播货盘。
- 加工厂和配套服务。

差异只体现在类型配置、品类、字段、展示和运营权重，不改变资源核心生命周期。

## 20. 数据一致性规则

### 20.1 状态一致性

资源状态、发布时间、过期时间、成交时间、下架时间、归档时间必须一致。

示例：

- `published` 应有 `published_at` 和 `expires_at`。
- `expired` 应有已到期的 `expires_at`。
- `dealt` 应有 `dealt_at`。
- `taken_down` 应有 `taken_down_at` 和必要的下架原因。
- `archived` 应有 `archived_at`。

### 20.2 公开查询一致性

所有公开入口必须使用同一套公开可见条件，不能出现：

- 搜索不可见但详情可访问。
- 首页不可见但专题可见。
- 过期资源在商家主页当前资源中展示。
- 已下架资源仍可联系。

### 20.3 审计数据不可删除

以下数据不得物理删除：

- 资源主记录。
- 审核记录。
- 联系事件。
- 效果统计。
- 认证记录。
- 信用记录。
- 操作日志。
- 撮合记录。

如需隐藏，应使用状态、软删除或撤销字段表达。

## 21. API 行为规则

### 21.1 用户端 API

用户端 API 应遵守：

- 默认只返回公开资源。
- 错误信息使用中文友好提示。
- ID 以字符串形式返回。
- 不返回内部审核备注、风控备注、数据库字段名和堆栈信息。
- 对无权限资源返回“不存在或已下架”类提示，避免泄露内部状态。

### 21.2 商家端 API

商家端 API 应遵守：

- 只允许查看和操作自己管理商家的资源。
- 我的资源列表可返回全部状态。
- 效果数据只对资源所属商家管理员和平台运营可见。
- 刷新、置顶、标记成交、下架必须校验状态和权限。

### 21.3 后台 API

后台 API 应遵守：

- 必须校验运营人员权限范围。
- 城市站运营只能管理授权城市站资源。
- 审核、下架、配置修改必须写操作日志。
- 返回数据可包含内部状态，但不得暴露敏感凭证或无关个人隐私。

## 22. 资源类型配置规则

资源类型配置包含：

- 类型编码。
- 类型名称。
- 所属城市站。
- 字段模板。
- 必填字段。
- 筛选字段。
- 展示模板。
- 审核规则。
- 排序权重。
- 消息规则。
- 默认有效期。
- 启用状态。

配置变更规则：

- 禁用资源类型后，不允许新建该类型资源。
- 禁用不应影响历史资源查看和后台审计。
- 修改必填字段只影响后续创建或重新提交，不应强制破坏已发布资源。
- 修改有效期只影响后续发布，是否影响已发布资源需由运营明确执行。
- 配置变更必须记录操作日志。

配置校验规则：

- `type_code` 必须稳定，不随中文名变化。
- `required_fields` 必须存在于通用字段或字段模板中。
- `filter_fields` 必须可被后端识别。
- `display_template` 不得引用不存在字段。
- `default_valid_days` 必须大于 0。

## 23. 反模式清单

必须避免以下做法：

- 为招聘、出租、服务商分别复制一套发布和审核系统。
- 为每个城市复制一套后台。
- 前端自行判断资源是否公开，后端不做状态过滤。
- 已过期、已下架资源仍出现在公开搜索结果。
- 置顶券让未审核资源提前公开。
- 认证通过后绕过资源审核。
- 审核驳回不写原因。
- 平台下架不写操作日志。
- 把曝光、浏览包装成真实成交。
- 为了快速扩业务不断新增表和页面，而不复用资源类型配置。

## 24. MVP 验收标准

资源管理 MVP 至少满足：

- 七类资源类型可通过统一资源模型发布。
- 资源类型配置可定义必填字段、筛选字段、展示模板和有效期。
- 商家或运营可以创建资源并提交审核。
- 待审核资源不会公开展示。
- 运营可以审核通过、驳回、下架资源。
- 已发布且未过期资源可以进入公开列表、搜索、详情和商家主页当前资源。
- 驳回、过期、成交、下架、归档资源不会进入公开结果。
- 商家可以查看自己的资源状态和基础效果数据。
- 已发布资源可以刷新、置顶、标记成交或下架，但必须受状态和权限约束。
- 联系动作可以记录电话点击、微信复制、商家主页访问或分享。
- 资源过期任务可以将到期资源标记为过期并发送消息。
- 审核记录、联系事件、效果统计和操作日志可追踪。

## 25. 后续演进方向

MVP 后可逐步增强：

- 收藏资源。
- 关注商家。
- 关注搜索条件。
- 新货提醒、库存更新提醒、工厂空档提醒。
- 更细的资源完整度评分。
- 更复杂的信用等级。
- 举报工单和申诉流程。
- 搜索排序实验。
- 城市站供需热词看板。
- 资源质量巡检任务。
- 成交反馈多方确认。

这些能力应继续围绕统一资源模型演进，避免破坏资源管理的公共能力边界。
