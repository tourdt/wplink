# wxapp 拿货地图页面设计

## 目标

参考《织里童装拿货导览地图产品开发文档》，在当前 `wxapp` 中新增一个独立的拿货地图页面，并先从 `我的`页提供入口。第一版目标不是建设完整地图平台，而是验证用户是否能从个人中心进入拿货导览，完成“看区域、找档口、筛选配套、查看详情”的核心体验。

本次采用小程序端 MVP 方案：新增页面、使用本地示例数据、实现轻量地图导览交互，暂不接入后端数据库、后台标注工具和真实档口批量数据。

## 当前系统判断

当前项目由三部分组成：

- `wxapp`：uni-app + Vue3 小程序，已有 `pages/my/index.vue` 作为“我的”页，已有 `pages.json` 统一注册页面和 tabBar。
- `backend`：Go/go-zero 风格后端，已有资源、商户、需求、收藏、消息等业务接口，但尚无 `map_scene`、`map_object`、`map_category` 相关表和接口。
- `admin-web`：后台管理端，已有资源审核、商户、需求等管理页面，但尚无地图底图上传、坐标标注、批量生成档口能力。

因此，完整产品文档中的第二阶段可用版会跨越小程序、后端和后台三端；如果一次性全部实现，范围过大且需要真实底图和数据录入流程支撑。当前最合适的切入点是小程序端独立页面 MVP。

## 方案比较

### 方案 A：静态导览页

只做一个图文导览页，展示织里热门街区、停车、物流、打包等信息，不做地图对象点击和搜索定位。

优点：

- 实现最快，主要是页面布局。
- 基本没有 Canvas 交互风险。
- 不依赖后端和真实坐标。

缺点：

- 不能验证产品文档里的核心“可交互拿货导览”价值。
- 后续迁移到真实地图页面时，很多页面结构需要重做。

适用场景：只想快速上线一个营销式入口，不验证地图能力。

### 方案 B：小程序端 MVP 地图页

新增独立拿货地图页面，使用本地 mock 数据定义场景、档口、POI 和分类，页面提供总览、搜索、筛选、对象高亮、详情抽屉等能力。地图区域先做轻量导览，不接后端和后台。

优点：

- 能验证核心体验：找区域、搜档口、筛配套、看详情。
- 数据字段可按产品文档的 `MapScene`、`MapObject`、`Category` 抽象设计，后续接 API 时迁移成本低。
- 改动集中在 `wxapp`，对现有后端和后台无侵入。

缺点：

- 只能代表交互原型，不能承载真实运营数据。
- 如果要做完整 Canvas 拖动缩放和命中检测，仍需真机验证。

适用场景：当前阶段先验证产品方向和页面体验。

### 方案 C：完整地图平台一期

同步建设后端表、API、小程序地图页、后台标注工具、底图上传和批量生成档口能力。

优点：

- 更接近产品文档中的第二阶段可用版。
- 数据维护闭环完整，能支撑真实上线运营。

缺点：

- 范围大，至少涉及数据库迁移、goctl 生成、后端逻辑、后台 Konva 标注、小程序 Canvas、底图和坐标数据。
- 没有稳定底图和首批真实档口数据时，开发风险高。
- 后台标注工具是独立复杂模块，不适合和入口页需求绑定一次完成。

适用场景：已经确认底图、数据来源、标注流程和运营人员投入。

## 推荐方案

推荐先做方案 B。

原因：

- 与当前请求“添加一个独立的拿货地图页面，先在我的页面添加入口”最贴合。
- 能把产品文档的核心模型提前体现在前端数据结构里。
- 改动范围可控，符合当前项目“精准改动、简洁优先”的规则。
- 可以在真实后端和后台建设前，先验证用户是否理解并使用这个入口。

## 范围

本次覆盖：

- `wxapp/pages/my/index.vue` 新增“拿货地图”入口。
- `wxapp/pages.json` 注册独立页面。
- 新增 `wxapp/pages/sourcing-map/index.vue` 页面。
- 新增本地示例数据文件，用于描述地图场景、档口对象、POI 对象和筛选项。
- 页面内实现搜索、筛选、对象选择、详情展示、搜索结果定位和高亮的轻量闭环。
- 调整 `wxapp/scripts/validate-pages.mjs`，把新页面纳入页面注册校验。
- 补充针对入口和页面注册的轻量测试。

本次不覆盖：

- 后端 `map_scene`、`map_object`、`map_category` 表和 API。
- `admin-web` 地图标注工具。
- 底图上传、档口批量生成、批量编号。
- 真实第三方地图导航唤起。
- 真实几百档口的大数据量渲染。
- 全量 Canvas 引擎、瓦片系统、文字避让、路径规划。

## 页面入口设计

在 `我的`页的常用服务列表中新增一个入口：

- 标题：`拿货地图`
- 副文案：`街区、档口、物流配套`
- 点击行为：`uni.navigateTo({ url: '/pages/sourcing-map/index' })`
- 登录要求：第一版不强制登录。拿货导览属于公共浏览能力，未登录用户也应该能查看；后续涉及收藏、商户认领、提交档口信息时再要求登录。

入口位置建议放在“收藏关注”之前或之后。它面向买家拿货，不属于商户管理动作；放在“收藏关注”附近更符合用户查找习惯。

## 页面结构

### 顶部搜索和区域提示

页面顶部展示：

- 搜索框，占位文案 `搜索档口、品类、打包、物流`
- 当前区域提示，例如 `织里核心拿货区`
- 简短状态文案，例如 `示例数据 · 可搜索女童/现货/打包`

搜索支持匹配：

- 档口编号。
- 档口名称。
- 主营分类名称。
- 服务标签名称。
- POI 名称。
- POI 类型。

### 快捷筛选

快捷筛选使用横向滚动按钮，第一版包含：

- 全部
- 女童
- 男童
- 中大童
- 现货
- 源头工厂
- 打包
- 物流
- 快递
- 停车

筛选后：

- 地图区只突出匹配对象。
- 非匹配对象可隐藏或降低透明度；第一版建议隐藏，逻辑简单。
- 页面显示结果数量，例如 `找到 8 个点位`。

### 总览和热门街区

页面保留“织里总览”区域，帮助新手用户理解拿货范围：

- 总览标题：`织里核心拿货区`
- 区域标签：`利济路`、`富民路`、`童装城周边`、`物流集中区`
- 热门街区卡片：`利济路中段`、`富民路中段`、`童装城周边`
- 每张卡片展示档口数、主营标签、配套摘要。

点击热门街区时，第一版切换当前场景或聚焦到对应区域；如果只保留一个示例场景，则可更新当前区域提示并高亮该区域相关对象。

### 地图导览区

第一版地图导览区采用“轻量 Canvas/伪地图”思路：

- 用固定比例容器模拟底图。
- 使用对象坐标渲染档口块和 POI 点位。
- 支持点击对象后高亮。
- 支持搜索结果点击后聚焦并选中对象。
- 支持基础缩放层级文案：远看区域，近看档口和配套。

如果实现 Canvas：

- 对象坐标基于示例 `scene.width` 和 `scene.height`。
- `rect` 用于档口。
- `point` 用于打包站、物流点、快递点、停车场。
- 点击命中优先判断 POI，再判断档口，避免点位被档口覆盖。

如果为了控制第一版风险，也可以先用绝对定位 `view` 渲染模拟地图；数据结构保持 `geometry` 不变，后续再替换为 Canvas。

### 详情抽屉

点击档口时展示：

- 档口编号。
- 档口名称。
- 主营分类。
- 服务标签。
- 地址。
- 电话和微信的示例展示。
- 附近配套：打包站、物流点、快递点、停车场。

点击 POI 时展示：

- 名称。
- 类型。
- 地址。
- 电话。
- 营业时间。
- 支持服务。
- 附近档口。

导航按钮第一版展示为禁用或提示文案：`真实导航待接入经纬度后开放`。这样避免用户误以为示例数据可以导航。

## 交互和状态

页面状态建议包含：

- `keyword`
- `activeFilter`
- `currentSceneCode`
- `selectedObjectId`
- `highlightObjectIds`
- `filteredObjects`
- `searchResults`
- `selectedObject`
- `nearbyPois`

核心规则：

- 搜索关键字为空时展示当前筛选结果。
- 输入关键字后展示搜索结果列表。
- 点击搜索结果后设置 `selectedObjectId` 和 `highlightObjectIds`。
- 点击地图对象后设置 `selectedObjectId`。
- 切换筛选项时，如果当前选中对象不在结果内，则清空选中对象。
- 附近配套第一版基于同一场景内 `centerX/centerY` 欧氏距离排序。

## 异常和兜底

- 无搜索结果时展示 `没有找到匹配点位`。
- 无筛选结果时展示 `当前分类暂无点位`。
- 对象缺少电话或微信时隐藏对应字段。
- 对象缺少地址时展示 `地址待完善`。
- 示例数据不提供真实导航，导航按钮只做提示。
- 页面不依赖登录态，接口失败风险不存在。

## 难度评估

入口和页面注册：低难度。

本地数据和搜索筛选：低到中等难度。

地图渲染和对象点击：中等难度。若使用绝对定位模拟地图，风险较低；若直接使用 Canvas 并支持拖动缩放，真机适配和命中检测风险会上升。

完整产品化：高难度。需要后端表、API、后台标注工具、底图规范、数据采集和运营审核流程。

## 风险分析

### 底图风险

真实底图尺寸一旦变化，坐标会整体失效。第一版必须把示例数据标明为演示数据，不承诺真实路线。

建议：正式接入前固定底图尺寸，避免随意裁剪和拉伸。

### 数据维护风险

真实织里档口数量大，靠手写 JSON 不可持续。

建议：MVP 验证通过后，下一阶段优先建设后台标注工具和批量生成能力，再接真实档口数据。

### 性能风险

小程序一次渲染几百个对象时，Canvas 和普通视图都会有性能压力。

建议：第一版控制在 20 个以内示例对象；后续真实场景按街区/路段拆分，每个 Scene 控制 200 到 500 个档口，并在低缩放层级隐藏文字。

### 认知风险

用户可能把示例页面误认为真实地图导航。

建议：页面顶部标明 `示例导览` 或 `试运行`，导航按钮提示真实导航待接入经纬度后开放。

## 验证标准

满足以下条件即可验收：

- `我的`页出现“拿货地图”入口。
- 点击入口可进入 `/pages/sourcing-map/index`。
- `pages.json` 正确注册页面，页面校验脚本通过。
- 页面展示搜索框、快捷筛选、总览区域、热门街区和地图导览区。
- 可以按关键字搜索档口或 POI。
- 可以按分类筛选档口或配套服务。
- 点击对象或搜索结果后，对象被高亮并展示详情。
- 点击档口后能展示附近配套列表。
- 无结果、缺少电话、缺少地址等情况有清晰兜底。
- 不修改后端、数据库和管理后台。
- `wxapp` 相关测试和构建通过。

## 后续演进建议

第一版验证通过后，建议按以下顺序推进：

1. 后端新增 `map_scene`、`map_object`、`map_category` 表和只读 API，使用 goctl 生成可生成代码。
2. 小程序页面从本地 mock 数据切换到 API 数据。
3. 后台新增场景管理、底图管理和矩形档口标注工具。
4. 后台增加批量生成档口、批量编号、批量标签能力。
5. 小程序补充真实第三方地图导航、真机 Canvas 拖动缩放优化和大数据量渲染优化。
