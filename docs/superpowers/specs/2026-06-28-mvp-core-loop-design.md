# MVP 核心闭环验证设计

日期：2026-06-28

## 目标

验证并补齐 MVP 最小业务闭环：

发布资源 -> 后台审核通过 -> 公开搜索和详情可见 -> 用户触发详情浏览和联系动作 -> 商家在我的发布中看到指标变化。

## 范围

本次只处理闭环直接需要的后端 API、数据流和必要的页面/API 调用缺口。不新增与闭环无关的运营功能，不扩展复杂推荐、交易、收藏、关注或营销能力。

## 验证路径

1. 创建或复用一个有效商家。
2. 创建一条资源草稿并提交审核，资源状态进入 `pending`。
3. 通过后台审核接口将资源置为 `published`，写入审核记录和必要消息。
4. 通过公开搜索接口确认资源可被检索到。
5. 通过公开详情接口确认资源详情可访问，且待审核、驳回、过期、下架资源不会出现在公开结果中。
6. 调用详情浏览接口记录一次浏览。
7. 调用联系事件接口记录电话和微信联系动作。
8. 查询资源指标或我的发布列表，确认详情浏览、电话点击和微信复制计数变化。

## 可能补缺点

- 路由未正确接入时，补接现有 logic，不新增平行实现。
- 状态流转或公开查询条件不一致时，以计划文档中的资源状态为准修正。
- 指标未聚合到我的发布或资源指标接口时，只补齐当前闭环需要的计数字段。
- 测试夹具不足时，优先补后端测试数据构造，不写入生产专用数据。

## 错误处理

所有前端可见错误保持中文、明确、可操作。后端日志可以保留内部原因，但响应不得暴露 SQL、表名、堆栈、token 或敏感原始字段。

## 验收标准

- 后端相关测试通过。
- 本地 API 可以复现完整闭环。
- `docs/superpowers/plans/2026-06-27-apparel-platform-current-mvp-todo.md` 的“第一个开发切片验证”可被更新为完成，或明确记录无法完成的环境原因。
