# 资源联系方式受控解锁设计

## 背景

衣货通的产品定位是服装产业带资源撮合平台，核心目标是帮助服装人更高效地找货、找厂、清库存和接订单。联系方式是资源撮合闭环的关键能力，不能只记录点击而不让用户真正联系到商家；同时，完整电话和微信也不应在公开详情接口中直接暴露，避免被批量爬取、搬运和骚扰。

当前资源公开详情只返回 `phoneMasked` 和 `wechatMasked`，资源详情页点击“复制微信”后先记录联系行为，再因为微信号仍是脱敏值而提示“已记录联系，完整微信由平台保护”。这个交互不符合用户对“复制微信”的预期，也会让商家侧 `wechatCopyCount` 混入未成功复制的行为。

## 产品原则

- 公开详情默认只展示脱敏联系方式。
- 用户点击电话或微信联系时，必须能在符合规则的情况下拿到完整联系方式。
- 联系成功才计入电话点击或微信复制指标。
- 解锁完整联系方式需要用户登录，但首期不强制绑定手机号。
- 手机号绑定用于增强验证、风控、商家管理和撮合回访，不作为普通买家首次联系的必填门槛。
- 登录手机号默认只用于账号安全和平台风控，不直接提供给被联系商家。
- 商家主页不直接暴露完整联系方式，仍引导用户从具体资源详情发起联系，保证联系行为有资源上下文。

## 登录与手机号规则

### 微信登录

资源联系方式解锁要求用户已通过微信登录获得平台账号和 token。微信登录只负责识别自然人账号，使用微信登录凭证换取后端 `userId` 和登录态。

未登录用户点击“拨打电话”或“复制微信”时：

- 前端跳转登录页，并携带当前资源详情页作为登录后回跳地址。
- 登录成功后回到资源详情页，用户可再次点击联系按钮。

### 手机号绑定

普通买家首次或低频联系资源时，不要求绑定手机号。

以下场景可触发手机号绑定：

- 发布资源。
- 提交采购需求。
- 申请商家认证或成为商家管理员。
- 单用户短时间高频复制微信或拨打电话。
- 账号命中风控规则。
- 平台运营需要对撮合或投诉进行回访。

手机号绑定是增强验证能力，不改变普通资源浏览和低风险联系链路。

## 联系方式解锁交互

资源详情页默认展示：

- 联系人：展示联系人姓名。
- 电话：展示脱敏电话，例如 `138****0000`。
- 微信：展示脱敏微信，例如 `zhil_****`。
- 底部按钮：`拨打电话`、`复制微信`、`分享`。

点击 `复制微信`：

1. 前端检查本地是否有 token。
2. 无 token 时跳转微信登录。
3. 有 token 时请求后端解锁微信联系方式。
4. 后端确认资源仍可公开联系，并记录成功解锁事件。
5. 后端返回完整微信号。
6. 前端调用剪贴板能力复制完整微信号。
7. 前端提示 `微信号已复制`。

点击 `拨打电话`：

1. 前端检查本地是否有 token。
2. 无 token 时跳转微信登录。
3. 有 token 时请求后端解锁电话联系方式。
4. 后端确认资源仍可公开联系，并记录成功解锁事件。
5. 后端返回完整电话号码。
6. 前端调用拨号能力。

如果商家没有填写微信，则提示 `商家暂未填写微信，可电话联系`，不计入微信复制指标。

如果商家没有填写电话，则提示 `商家暂未填写电话，可查看商家主页或联系平台`，不计入电话点击指标。

## 后端接口设计

公开详情接口继续只返回脱敏联系方式：

- `GET /api/v1/resources/{resourceId}`
- 返回 `contact.phoneMasked` 和 `contact.wechatMasked`。
- 不返回完整 `phone` 或 `wechat`。

联系事件接口调整为受控解锁接口：

- `POST /api/v1/resources/{resourceId}/contact-events`
- 请求体：`{ "action": "wechat" }` 或 `{ "action": "phone" }`。
- 必须携带用户 token。
- 后端从 token 解析 `userId`，不接受前端传入 `userId` 作为归因身份。
- 成功时返回对应完整联系方式。

建议响应示例：

```json
{
  "message": "微信号已解锁",
  "action": "wechat",
  "wechat": "zhili_stock_demo"
}
```

```json
{
  "message": "电话已解锁",
  "action": "phone",
  "phone": "13800000000"
}
```

`merchant_home`、`merchant_profile` 和 `share` 可以继续作为联系事件记录，但不返回联系方式，也不要求手机号绑定。

## 后端校验规则

后端必须作为最终判断方，不能只依赖前端状态。

联系方式解锁时必须校验：

- 用户已登录。
- 资源存在。
- 资源状态为 `published`。
- 资源未过期、未下架、未归档、未删除。
- 所属商家仍可正常展示。
- 当前用户不是该资源所属商家的管理员。
- 对应联系方式存在且非空。

失败时使用稳定中文错误：

- 未登录：`请先登录后联系商家`。
- 资源不可联系：`资源不存在或已下架`。
- 自己的资源：`不能联系自己发布的资源`。
- 微信缺失：`商家暂未填写微信，可电话联系`。
- 电话缺失：`商家暂未填写电话`。
- 高频限制：`联系过于频繁，请稍后再试`。

## 指标与事件规则

联系方式成功解锁后才写入对应指标：

- `phone` 成功返回完整电话后，增加 `phone_click_count`。
- `wechat` 成功返回完整微信后，增加 `wechat_copy_count`。
- `merchant_home` 或 `merchant_profile` 增加联系总数，但不增加电话或微信指标。
- `share` 增加分享指标。

事件表仍记录原始联系事件，便于后续审计和风控。

商家侧指标可以保留总次数；后续可增强为：

- 总联系次数。
- 独立联系人数。
- 电话点击人数。
- 微信复制人数。
- 当日趋势。

MVP 阶段至少保证未成功解锁的行为不计入电话或微信成功指标。

## 风控边界

首期不做复杂风控，但接口设计需要保留扩展点。

建议基础限制：

- 同一用户对同一资源同一联系方式短时间重复点击，只记录事件，不重复增加成功指标。
- 单用户每日联系方式解锁数量超过阈值后，要求绑定手机号或临时限制。
- 匿名用户不能解锁完整联系方式。
- 被平台封禁或异常状态用户不能继续解锁。

这些规则可先在后端逻辑中保留清晰判断边界，阈值和 Redis 分布式限流可后续补齐。

## 前端调整范围

资源详情页需要调整：

- 点击联系前检查登录态。
- 未登录时跳转登录页并带回跳地址。
- `copyWechat()` 使用后端返回的完整 `wechat` 写入剪贴板。
- `callPhone()` 使用后端返回的完整 `phone` 发起拨号。
- 成功复制微信时提示 `微信号已复制`。
- 不再出现“已记录联系，完整微信由平台保护”这种与按钮预期冲突的提示。
- 联系失败时展示后端中文错误。

商家主页保持现有策略：

- 不直接展示完整联系方式。
- 引导用户从具体资源详情联系商家。

## 不做范围

本设计不包含：

- 即时通讯。
- 担保交易、支付、下单、合同和履约。
- 把登录手机号直接提供给商家。
- 商家主页直接复制完整联系方式。
- 复杂信用分。
- 完整反爬系统。

## 验收标准

满足以下条件即可验收：

- 未登录用户点击资源详情电话或微信联系时，会被引导登录。
- 已登录用户点击“复制微信”后，剪贴板拿到完整微信号，并提示 `微信号已复制`。
- 已登录用户点击“拨打电话”后，使用完整手机号发起拨号。
- 公开详情接口仍只返回脱敏联系方式。
- 商家未填写微信时，不计入 `wechatCopyCount`。
- 商家未填写电话时，不计入 `phoneClickCount`。
- 用户不能联系自己发布的资源并刷指标。
- 已过期、已下架、未发布资源不能解锁联系方式。
- 商家“我的发布”中电话点击和微信复制指标只统计成功解锁行为。
- 手机号绑定不阻断普通买家的低风险首次联系。
